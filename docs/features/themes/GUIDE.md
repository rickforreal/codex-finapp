# Theme Guide (FinApp)

This guide explains how themes work in FinApp and how to add a new built-in theme safely.

## 1) Theme System Overview

FinApp uses a **server-defined theme model**:

1. Server owns theme definitions and exposes them via `GET /api/v1/themes`.
2. Client loads theme payload on startup.
3. Client resolves active theme with precedence:
   - snapshot-selected family/appearance (if valid)
   - local browser preference
   - server default selection
4. Client applies:
   - primitive token CSS vars (existing `--theme-color-*`, `--theme-chart-*`, etc.)
   - resolved slot CSS vars (`--theme-slot-*`) for granular component styling

Theme selection model:
- `ThemeFamilyId` (e.g. `default`, `monokai`, `stayTheCourse`)
- `ThemeAppearance` (`light | dark`)
- Active render theme is a concrete `ThemeVariantId` (e.g. `default.light`).

## 2) Token Model (v2)

Theme payload now includes:

- `tokenModelVersion: "2"`
- `defaultSelection`
- `families` (grouped selectors + supported appearances)
- `variants` (concrete theme definitions)
- `tokens` (primitives): color/typography/spacing/radius/border/shadow/motion/state/chart
- `semantic`: global alias map
- `slots`: component-slot alias map
- `overrides` (optional): per-theme slot overrides
- `slotCatalog`: canonical slot registry with fallback references

Compatibility aliases remain during migration:
- `defaultThemeId`
- `themes`
- `catalog`

Resolution order for each slot is deterministic:

1. `theme.overrides[slotPath]`
2. `theme.slots[slotPath]`
3. `slotCatalog.fallback`
4. semantic alias
5. primitive token value

## 4) How to Add a New Theme

### Step 1: Add Theme Family/Variant IDs

Add enum entry in:
- `packages/shared/src/constants/enums.ts` (`ThemeFamilyId`, `ThemeVariantId`, optionally `ThemeId` legacy alias)

### Step 2: Add Theme Variants in Server Registry

In `packages/server/src/themes/registry.ts`, add:

- family entry (`id`, display metadata, `supportedAppearances`)
- one or more concrete variant entries (`familyId + appearance + tokens`)

`semantic` and `slots` are generated by registry mapping logic for v2.
Use `overrides` only when this theme needs explicit slot deltas beyond defaults.

### Step 3: Validate Slot/Token Integrity

Server validation already checks:

- unresolved semantic/slot refs
- unknown slot keys
- key contrast-sensitive pairs

If warnings appear, adjust token values and rerun tests.

### Step 4: Verify Client Rendering

Launch app and switch to the new theme from the command bar theme menu.
Smoke-test these surfaces:

1. Command bar (buttons, popovers, modal, tooltips)
2. Sidebar and section cards
3. Input controls
4. Summary cards
5. Chart panel + chart tooltips
6. Detail ledger states (focus/edited/stale)
7. Stress panel

### Step 5: Run Regression Gate

From repo root:

```bash
npm run typecheck
npm run lint
npm test
npm run build
```

## 5) Adding/Changing Slot Coverage

If a UI element is not granularly styleable yet:

1. Add/update slot in `defaultSlotMap` (server registry).
2. Ensure `slotCatalog` entry exists (auto-derived from map).
3. Add/adjust client CSS class that consumes `--theme-slot-*`.
4. Apply class in relevant component.
5. Add/update tests if behavior changed.

## 6) Compare Slot Color Quality Bar

Compare slot tokens (`ThemeChartTokens.compareSlotA..H`) drive:
- sidebar compare chips
- compare tabs
- compare chart line colors

When creating a theme, ensure these colors are intentionally distinct from each other in both appearances.
Do not constrain compare-slot tokens to only narrow shades of one hue if that reduces visual separation.
It is acceptable (and recommended when needed) to introduce additional accent colors beyond the base brand palette to maintain line/chip distinguishability and readability.

## 7) Snapshot/Bookmark Compatibility

Theme persistence remains by selected family/appearance and cached theme payload.
Snapshot parsing tolerates missing `theme.slotCatalog` by defaulting to `[]` for backward compatibility.

## 8) Troubleshooting

If a slot does not appear to update:

1. Confirm slot path exists in server `defaultSlotMap`.
2. Confirm payload includes expected `slotCatalog` and theme maps.
3. Confirm client class references correct `--theme-slot-<kebab-path>`.
4. Check for hardcoded utility classes overriding slot styles.
5. Check `validationIssues` in `/themes` payload and browser console warnings.
